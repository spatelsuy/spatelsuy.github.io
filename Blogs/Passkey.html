How Passkey Works: A Step-by-Step Walkthrough with Windows Hello and Trusted Platform Module (TPM)

The scenario is: I have a personal laptop for home use. At this point I am the only user of the laptop, having a local admin account “accountA”. Recently bank (bank.com) has mandated to use Passkey replacing traditional username and password.

Curious about how the process works end-to-end, from passkey registration to actual login? What happens behind the scenes with Windows Hello, the Trusted Platform Module (TPM), and the browser?

There are two phases:

  1. Registration Phase – where the passkey is created.
  2. Authentication Phase – where the passkey is used to log in.

Registration Flow

  1. I have Windows laptop with a local account “accountA”.
  2. I have a bank account in bank (bank.com).
  3. I laptop is having latest Chrome browser (not signed into a Google account) to access the bank site bank.com.
  4. The site prompts to log in using a passkey, and I click Yes.
  5. Since no passkey is set up yet, Chrome initiates the creation of a new passkey.
  6. Chrome does not create the passkey itself, it uses the WebAuthn API to interact with the platform authenticator, in this case it is Windows Hello.
  7. The WebAuthn request to Windows Hello triggers the following actions:

    • Windows Hello calls the TPM API to generate a key pair (private + public key), where:
      • The private key is securely created and stored inside the TPM.
      • The private key never leaves the TPM and is only usable via TPM APIs.
    • Credential ID (a random handle) is generated by Window Hello to reference this key pair.
    • The Credential ID and key pair are scoped (bound) to the Relying Party (RP): bank.com.
    • Windows Hello prompts the user for biometric verification (e.g., fingerprint or face ID).
    • Assume I use fingerprint.
    • Windows Hello captures the biometric and generates a non-reversible hash (template).
    • This hashed biometric template is stored securely in the Windows Hello store, not in the TPM and not linked to any RP or Credential ID.
    • Biometrics are used only for user verification (UV), not for signing or credential selection.
  8. After successful verification, Windows Hello returns the public key and Credential ID to Chrome.
  9. Chrome forwards the public key and Credential ID to the bank backend.
  10. The bank stores them, linking this passkey to the user account for future authentication.
  11. Registration is complete. I can now log in to bank using the passkey, without a password.
