Token in IAM

Let’s try to understand it in a simple way.

Imagine you visit a bank to withdraw money. You fill out a withdrawal form by providing necessary details like your account number and the withdrawal amount. You sign the form and hand it over to the clerk. The clerk verifies your identity (you need to show an identity card) and matches the signature to ensure that you are the rightful account holder. (This process is called request and verification.)

Instead of directly handing over the money, the clerk gives you a token with a number - this represents an access token in the digital world.

You then approach the cashier and present the token. At this point, the cashier does not verify your identity again; instead, they match the token number with the details already received from the clerk. After verifying the token, the requested amount is handed over to you.

Note: There is a security flaw in this process. Anyone possessing the token can approach the cashier to withdraw the money, because the cashier trusts only the token, not your identity.

Similarly, in the digital world, an access token grants access without requiring re-authentication. If a hacker gets hold of your access token, they can access services as you — without needing your credentials.

Understanding this analogy highlights the importance of token security - tokens must be kept safe and are often short-lived to minimize risk.

 

Quick Definitions:

  • Access Token: A credential issued by the Authorization Server after successful authentication and authorization. It allows a client to access protected resources on behalf of the user. These tokens are typically short-lived for security reasons.
  • ID Token: Used in OpenID Connect (OIDC). It contains user identity information like name, roles, and authentication timestamp. It is used by client applications to identify the authenticated user.
  • Refresh Token: A long-lived token used to obtain new access tokens without requiring the user to log in again. Used in OAuth 2.0, it is stored securely on the client-side and not sent to APIs, but exchanged with the Authorization Server.
  • JWT (JSON Web Token): A compact, self-contained format for encoding user information or authorization data. JWTs are commonly used for ID Tokens and Access Tokens in OAuth 2.0, OIDC, and Single Sign-On. They can be verified locally using a public key, without needing to call the Authorization Server.
  • Bearer Token: A method of sending an access token in API requests. The term implies that the "bearer" (whoever holds the token) is authorized to access protected resources without further authentication.
